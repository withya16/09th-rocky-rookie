# 1. 파일 시스템의 설계와 데이터 보존

## 1-1. RHEL 표준 파일 시스템 (XFS & Ext4)

### 도입 배경

- Ext4는 안정적이지만 수백 TB 이상의 대규모 스토리지에서는 확장성에 한계를 보임
  <br/>-> 고성능 병렬 처리에 최적화된 XFS를 기본 파일 시스템으로 채택

### Ext4

- 중소규모 서버에 적합
- 강점 : 파일 시스템 크리를 줄이는 기능

### XFS

- 64비트 주소 체계 사용 -> 거대한 파일과 파티션 지원
- 강점 : 병렬 I/O 성능이 좋음 (크리 축소 불가능)

## 1-2. Inode 구조와 B+ Tree 인덱싱

### Inode (Index Node)

- 파일의 이름만 제외하고 모든 정보(소유자, 권한, 크기, 데이터 블록의 위치)를 담고 있는 메타데이터 주머니

### B+ Tree 인덱싱

- 파일이 수백만 개가 되어도 원하는 파일을 빛의 속도로 찾기 위한 색인 기술
- XFS에서 디렉토리와 비어 있는 공간 관리에 모두 사용

## 1-3. Allocation Group(AG)

### 도입 이유

- 일반적인 파일 시스템은 디스크의 앞부분부터 차례로 쓰려고 함
  <br/>-> 여러 프로세스가 동시에 데이터를 쓰려 하면 느려짐
  <br/>=> 전체 파일 시스템을 여러 개의 독립적인 구역으로 분리

### 동작 방식

- 각 AG는 자신만의 Inode와 프리 스페이스 관리를 따로 진행
  <br/>-> 동시에 데이터 기록 가능

## 1-4. 저널링(Journaling) 메커니즘

### 도입 배경

- 데이터를 쓰는 도중 전우너이 나가면 데이터는 저장되었는데 로그로는 기록되지 않음

### 동작 원리

1. Journal(Log) 기록

- 실제 데이터를 디스크에 쓰기 전, 어느 영역에 작업할지 계획을 파일 시스템의 저널 영역에 미리 작성

2. 실제 쓰기

- 계획대로 데이터를 실제 위치에 작성

3. 완료 마크

- 쓰기가 끝나면 저널에 표기

4. 복구 시

- 사고 발생 시, 저널만 관찰
  - '완료' 표시가 없는 계획만 찾아서 다시 실행하거나 취소하면 종료
    <br/>-> 전체 디스크를 뒤질 필요가 없어 복구가 수초 내에 종료

# 2. VFS 추상화와 마운트 메커니즘

## 2-1. VFS(Virtual File System)

### 동작 원리

- 사용자가 read()나 write()라는 표준 시스템 콜을 던지면, VFS가 이를 받아서 해당 디스크가 XFS인지 Ext4인지 확인한 후, 그에 맞는 전용 함수로 변환하여 전달

### 핵심 객체

- Superblock: 파일 시스템 전체의 요약 정보
- Inode: 개별 파일의 정보
- Dentry (Directory Entry): 파일 이름과 Inode를 연결해주는 캐시

## 2-2. 마운트(Mount)와 네임스페이스

### 마운트(Mount)

- 특정 디렉터리(마운트 포인트)를 외부 장치의 입구로 지정하는 행위
- 마운트가 완료되면 해당 디렉터리로 들어가는 모든 요청은 실제 디스크 장치로 전달

### 네임스페이스(Mount Namespace)

- 현대 리눅스(컨테이너 기술의 핵심)에서는 프로세스마다 서로 다른 '마운트 지도' 보유 가능

## 2-3. 장치 점유(Busy)와 Lazy Unmount

### 도입 이유

- 디스크를 빼려고 했을 때 `device is busy` 에러가 나는 상황을 대비
  - `device is busy` : 누군가 그 안에서 파일 열람 중

### 해결 원리

1. Busy 상태 확인

- `lsof`나 `fuser` 명령어로 어떤 프로세스가 원인인지 발견 가능

2. Lazy Unmount (`umount -l`)

- 동작: 즉시 시스템의 디렉터리 트리에서 해당 장치를 분리(Detach)
- 결과
  - 새로운 프로세스는 더 이상 그 디스크에 접근 불가능
  - 이미 사용 중인 프로세스가 작업을 마칠 때까지 커널은 내부적으로 장치를 완전히 해제하지 않고 기다렸다가, 마지막 사용자가 사라지는 순간 조용히 정리

# 3. 스토리지 가상화 및 입출력 최적화

## 3-1. 페이지 캐시(Page Cache)의 동작

### Read Cache
![page cache](image/image_0401.png)

- 한 번 읽은 데이터를 버리지 않고 커널 메모리의 남는 공간(Page Cache)에 할당
- 다음에 똑같은 데이터를 요청하면 디스크로 가지 않고 메모리에서 즉시 반환

### Dirty Page

- 데이터를 쓸 때도 일단 메모리에 먼저 작성
- 디스크와 내용이 달라진 메모리 영역을 'Dirty(더럽혀진) 페이지'라고 지칭
- 커널은 나중에 한꺼번에 이들을 디스크에 전(Flush).

## 3-2. Direct I/O vs Buffered I/O

| 구분 | Buffered I/O (기본값)                      | Direct I/O (O_Direct)                  |
| ---- | ------------------------------------------ | -------------------------------------- |
| 경로 | 앱 ↔ 페이지 캐시 ↔ 디스크                  | 앱 ↔ 디스크 (직통)                     |
| 장점 | 작은 데이터를 자주 다룰 때 압도적으로 빠름 | 커널 메모리 복사 비용(Overhead)이 없음 |
| 단점 | 메모리 부족 시 캐시 정리 비용 발생         | 앱이 스스로 캐시 관리 필요 (매우 복잡) |

## 3-3. 지연 할당(Delayed Allocation)

### 동작 방식

1. 할당 보류

- 애플리케이션이 데이터를 저장하겠다고 하면, 파일 시스템(XFS, Ext4)은 즉시 디스크 주소를 할당하지 않고 메모리에만 데이터 저장

2. 최적의 위치 탐색

- 데이터가 충분히 모일 때까지 기다렸다가(보통 수 초), 전체 데이터 크기를 확인

3. 일괄 기록

- 그 크기에 딱 맞는 커다란 빈 공간을 디스크에서 한 번에 찾아내어 연속적으로 저장
