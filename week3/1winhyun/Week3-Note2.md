# RHEL 10 기반 마운트, LVM, 파일 시스템 관리 및 자동 마운트 학습 노트

> RHEL 10 기반의 리눅스 스터디를 위한 마운트 개념, LVM 스토리지 구성, 파일 시스템 관리 및 자동 마운트 실습에 대한 상세한 학습 자료입니다. 실무와 자격증(RHCSA 등) 준비에 바로 활용하실 수 있도록 구성했습니다.

---

## 1. 마운트(Mount)의 개념

리눅스 및 유닉스 계열 운영 체제에서 **마운트(Mount)** 란 물리적인 저장 장치(하드 디스크, 파티션, USB 등)에 생성된 파일 시스템을 리눅스의 기존 디렉터리 트리 구조의 특정 지점(마운트 포인트)에 '연결'하는 과정을 말합니다.

### 왜 마운트가 필요한가?

윈도우에서는 디스크를 추가하면 `D:`, `E:` 와 같이 독립적인 드라이브 문자가 할당되지만, 리눅스는 모든 파일과 디렉터리가 최상위 `/`(루트) 디렉터리에서 시작하는 **단일 트리 구조**를 가집니다. 따라서 새로운 디스크나 파티션을 사용하려면 이 트리 구조의 특정 빈 디렉터리(예: `/mnt/data` 또는 `/usr`)에 해당 저장 장치를 논리적으로 연결해야만 OS와 사용자가 해당 장치의 데이터에 접근할 수 있습니다.

### 예시

`/dev/sda5` 파티션을 `/usr` 디렉터리에 마운트했다면, `/usr` 아래에 있는 모든 파일과 디렉터리는 물리적으로 `/dev/sda5` 디스크 영역에 저장됩니다.

## 2. LVM (Logical Volume Manager)의 개념과 볼륨 구성

LVM은 물리적 디스크 스토리지를 직접 사용하는 대신, 그 위에 추상화된 가상 계층을 만들어 유연성을 제공하는 **스토리지 가상화 기술**입니다. 디스크 파티션의 크기를 시스템 가동 중(온라인)에 동적으로 확장하거나 축소할 수 있게 해줍니다.

### LVM의 3단계 구조

#### PV (Physical Volume - 물리 볼륨)

물리적인 디스크(예: `/dev/sdb`)나 파티션을 LVM이 사용할 수 있도록 메타데이터를 기록해 초기화한 상태입니다.

```bash
# 실습 명령어
pvcreate /dev/sdb
```

#### VG (Volume Group - 볼륨 그룹)

하나 이상의 PV를 모아 만든 거대한 가상의 스토리지 풀(Pool)입니다. VG는 **PE(Physical Extents)**라는 일정한 크기의 블록(RHEL 10 기본값: 4MiB)으로 나뉘며, 이 PE들이 모여 실제 용량을 구성합니다.

```bash
# 실습 명령어
vgcreate myvg /dev/sdb /dev/sdc
```

#### LV (Logical Volume - 논리 볼륨)

VG의 용량을 떼어내어 사용자가 실제로 파일 시스템을 생성하고 마운트할 수 있도록 만든 최종 가상 파티션입니다.

```bash
# 실습 명령어 (예: 10GB 볼륨 생성)
lvcreate --name mylv --size 10G myvg

# 온라인 확장 (-r 옵션을 주면 파일 시스템 크기까지 한 번에 자동으로 늘려줍니다)
lvextend -r -L +5G /dev/myvg/mylv
```

### LVM 스냅샷 (Snapshot)

LVM 스냅샷은 특정 시점의 논리 볼륨(LV) 상태를 그대로 보존하는 복사본입니다. 전체 데이터를 통째로 복제하는 것이 아니라 **CoW(Copy-on-Write)** 방식을 사용합니다. 스냅샷 생성 시점에는 원본 데이터를 가리키는 포인터만 만들고, 이후 원본 데이터가 변경될 때만 변경 전 데이터를 스냅샷 영역에 복사하여 보존합니다. 따라서 스냅샷 생성이 거의 즉각적이며, 할당하는 용량도 원본보다 훨씬 작게 설정할 수 있습니다.

#### 스냅샷의 동작 원리 (CoW)

1. **스냅샷 생성 직후**: 원본 LV와 동일한 데이터를 공유하며, 별도의 데이터 복사는 발생하지 않습니다.
2. **원본 데이터 변경 시**: 변경되기 직전의 원본 블록을 스냅샷 영역에 먼저 복사한 뒤, 원본을 덮어씁니다.
3. **스냅샷 읽기 시**: 변경된 블록은 스냅샷 영역에서, 변경되지 않은 블록은 원본에서 읽어 스냅샷 생성 시점의 상태를 재현합니다.

> **주의:** 스냅샷 영역의 용량이 가득 차면(원본 변경량이 스냅샷 크기를 초과) 스냅샷이 무효화(invalid)됩니다. 변경량이 많을 것으로 예상되면 충분한 크기를 할당해야 합니다.

#### 주요 용도

- **백업**: 서비스를 중단하지 않고 일관된 시점의 데이터를 백업할 수 있습니다. 스냅샷을 마운트하여 백업한 뒤 삭제하면 됩니다.
- **롤백**: 시스템 업데이트나 패치 전에 스냅샷을 생성해두고, 문제 발생 시 `lvconvert --merge`로 이전 상태로 복원합니다.
- **테스트**: 운영 데이터의 스냅샷을 별도로 마운트하여, 원본에 영향 없이 안전하게 테스트할 수 있습니다.

#### 실습 명령어

```bash
# 스냅샷 생성 (원본: /dev/myvg/mylv, 스냅샷 크기: 1GB)
lvcreate --snapshot --name mylv_snap --size 1G /dev/myvg/mylv

# 스냅샷 상태 확인 (사용률 등)
lvs

# 스냅샷을 별도 디렉토리에 마운트하여 백업/확인
mkdir -p /mnt/snap
mount -o ro /dev/myvg/mylv_snap /mnt/snap

# 스냅샷으로 원본 복원 (복원 후 스냅샷은 자동 삭제됨)
umount /mnt/snap
lvconvert --merge /dev/myvg/mylv_snap

# 스냅샷이 불필요할 경우 삭제
lvremove /dev/myvg/mylv_snap
```

#### 스냅샷, 백엔드 개발에서도 쓰인다. 계층은 달라도 철학은 같다

LVM 스냅샷을 학습하면서 백엔드 프로젝트에서 적용했던 스냅샷 패턴이 떠올랐습니다. 메뉴 관리 서비스를 개발할 때, 메뉴가 수정될 때마다 기존 데이터를 변경하지 않고 새로운 스냅샷 행(row)을 생성하여 변경 이력을 보존하는 구조를 설계한 적이 있습니다. 예를 들어 주문 시점의 메뉴 이름과 가격이 이후에 바뀌더라도, 스냅샷 테이블에 당시 상태가 그대로 남아있어 정확한 주문 내역을 조회할 수 있었습니다.

두 스냅샷은 적용 계층이 전혀 다릅니다:

| 구분 | LVM 스냅샷 | 애플리케이션 스냅샷 |
|------|-----------|-------------------|
| **계층** | OS/인프라 (디스크 블록) | 애플리케이션 (데이터베이스 행) |
| **대상** | 논리 볼륨의 전체 블록 데이터 | 엔티티의 비즈니스 데이터 (이름, 가격 등) |
| **방식** | CoW로 변경 전 블록을 별도 영역에 보존 | 업데이트 시 새로운 불변(immutable) 행 생성 |
| **복원** | `lvconvert --merge`로 볼륨 롤백 | 과거 스냅샷 행을 조회하여 시점별 상태 확인 |

그러나 핵심 철학은 동일합니다. **"기존 데이터를 수정하지 않고, 특정 시점의 상태를 별도로 보존한다"** 는 불변성(Immutability) 원칙이 인프라 계층과 애플리케이션 계층 모두에서 동일하게 활용되고 있음을 확인할 수 있었습니다.

### RHEL 9부터 도입된 LVM 장치 관리 방식 (system.devices)

> 과거에는 정규 표현식 기반의 LVM 필터를 사용했으나, RHEL 9부터 `/etc/lvm/devices/system.devices`라는 **명시적인 화이트리스트 레지스트리**를 기본적으로 사용합니다. LVM은 이 파일에 등록된 장치만 스캔하고 접근하므로, 복잡한 SAN/멀티패스 환경에서 발생하던 "중복 PV" 경고나 설정 오류를 근본적으로 방지하여 보안과 안정성이 대폭 강화되었습니다.

## 3. 파일 시스템 생성 및 점검 (XFS vs ext4)

LVM으로 논리 볼륨(LV)을 만들었다면, 데이터를 저장할 수 있도록 포맷(파일 시스템 생성)을 해야 합니다. RHEL 10에서는 주로 **XFS**와 **ext4**를 사용합니다.

### XFS (RHEL 10 기본 파일 시스템)

- **특징**: 대규모 데이터와 높은 동시성 처리에 최적화된 64비트 저널링 파일 시스템입니다. RHEL 10 기준으로 파일 시스템 최대 1024 TiB, 단일 파일 최대 8 Exabyte를 지원합니다.
- **장점**: '할당 그룹(Allocation Groups)' 구조를 통해 다중 스레드 I/O를 병렬로 처리하므로 대용량 파일과 데이터베이스에 매우 빠릅니다. 또한 데이터 중복 없이 파일 복제본을 즉시 생성하는 **reflink(Copy-on-Write)** 기능도 지원합니다.
- **주의점**: 온라인 확장은 가능하지만, 파일 시스템의 크기를 **축소(Shrink)하는 것은 불가능**합니다. 축소가 필요하다면 데이터를 백업하고 볼륨을 완전히 다시 만들어야 합니다.

```bash
# 생성
mkfs.xfs /dev/myvg/mylv

# 점검(복구) - 파일 시스템이 언마운트된 상태에서 실행
xfs_repair /dev/myvg/mylv
```

### ext4 (안정적인 범용 파일 시스템)

- **특징**: 가장 오랫동안 검증된 리눅스 표준 파일 시스템으로 최대 50 TiB의 크기를 지원합니다.
- **장점**: XFS와 달리 오프라인 상태에서 파일 시스템 크기를 **축소할 수 있는 유연성**을 제공하므로, 용량 회수가 필요할 수 있는 파티션에 유리합니다. 자잘한 파일이 매우 많은 환경에서도 오버헤드가 적어 좋은 성능을 냅니다.

```bash
# 생성
mkfs.ext4 /dev/myvg/mylv

# 점검(복구)
e2fsck -f /dev/myvg/mylv
```

## 4. 마운트 및 /etc/fstab 자동 마운트 설정 실습

임시 마운트는 `mount` 명령어로 수행하지만, 재부팅 시에도 영구적으로 마운트가 유지되려면 반드시 **`/etc/fstab`** 파일에 기록해야 합니다.

### 영구적 이름 지정 (UUID 사용의 중요성)

디스크는 부팅 순서나 포트 변경에 따라 이름이 `/dev/sdb`에서 `/dev/sdc`로 변할 수 있습니다. 따라서 RHEL 10은 하드웨어 변화에 영향을 받지 않는 영구 식별자인 **UUID**를 사용하여 마운트할 것을 강력히 권장합니다.

```bash
# UUID 확인 명령어
lsblk --fs
blkid
```

### /etc/fstab 설정 방법

`/etc/fstab` 파일은 공백(스페이스 또는 탭)으로 구분된 총 **6개의 필드**로 구성됩니다.

```
[디바이스/UUID]  [마운트포인트]  [파일시스템타입]  [마운트옵션]  [Dump옵션]  [fsck체크순서]
```

### 실습 예제: 새 데이터 볼륨 추가하기

1. **UUID 확인**: `lsblk --fs /dev/myvg/mylv` 를 통해 `1cdfbc...` 와 같은 UUID 값 확인
2. **마운트 포인트(디렉터리) 생성**: `mkdir -p /mnt/data`
3. **`/etc/fstab` 파일 편집**하여 아래 줄 추가:

```
UUID=1cdfbc...  /mnt/data  xfs  defaults,nofail  0  2
```

### 실습 예제: 스왑 메모리 할당 및 영구 등록

스왑(Swap)은 물리적 RAM이 부족할 때 디스크의 일부를 임시 메모리처럼 사용하는 공간입니다. `swapon`으로 활성화한 스왑은 재부팅 시 해제되므로, `/etc/fstab`에 등록하여 영구적으로 유지해야 합니다.

1. **스왑 파일 생성** (2GB 예시):

```bash
sudo dd if=/dev/zero of=/swapfile bs=128M count=16
```

2. **권한 설정** (root만 읽기/쓰기 가능하도록 제한):

```bash
sudo chmod 600 /swapfile
```

3. **스왑 영역으로 포맷**:

```bash
sudo mkswap /swapfile
```

4. **스왑 활성화 및 확인**:

```bash
sudo swapon /swapfile
sudo swapon -s
free -h
```

5. **`/etc/fstab` 파일 편집**하여 아래 줄 추가:

```
/swapfile  swap  swap  defaults  0  0
```

> **fstab 필드 해석**: 마운트 포인트와 파일 시스템 타입이 모두 `swap`이며, fsck 순서는 `0`입니다. 스왑은 일반 파일 시스템이 아니므로 디렉토리에 마운트하지 않고, 파일 시스템 검사도 불필요합니다.

### 주요 필드 및 권장 옵션 설명

#### 마운트 옵션 (Options)

| 옵션 | 설명 |
|------|------|
| `defaults` | `rw`, `suid`, `dev`, `exec`, `auto`, `nouser`, `async` 등의 기본값이 적용됩니다 |
| `nofail` | 외장 하드나 비핵심 데이터 디스크에 필수적인 옵션입니다. 부팅 시 해당 디스크가 없거나 마운트가 실패해도 시스템 부팅을 멈추지 않고 계속 진행하게 해줍니다 |

#### 보안 강화를 위한 옵션 (Security Options)

| 옵션 | 설명 |
|------|------|
| `noexec` | 해당 파티션에 있는 바이너리 파일의 실행을 차단합니다 (데이터 전용 볼륨에 설정 권장) |
| `nosuid` | 특수 권한(SUID/SGID)이 동작하지 않도록 하여 권한 상승 취약점을 막습니다 |
| `nodev` | 장치 파일(Character/Block 디바이스)의 인식을 차단합니다 |

#### 파일 시스템 검사 순서 (fsck order - 6번째 필드)

| 값 | 설명 |
|----|------|
| `1` | 최상위 루트(`/`) 파일 시스템에만 설정합니다 |
| `2` | 루트를 제외한 나머지 모든 디스크 파티션에 설정합니다 |
| `0` | 스왑(Swap) 파티션, 네트워크 파일 시스템(NFS 등) 또는 검사를 건너뛸 볼륨에 설정합니다 |

### 매우 중요한 마무리: 적용 및 검증

> **`/etc/fstab`에 오타가 있거나 설정이 잘못되면 시스템이 부팅 불능(Emergency mode) 상태에 빠집니다.** 따라서 파일을 수정한 후 재부팅하기 전에 반드시 아래 명령어로 문법 및 마운트 오류가 없는지 테스트해야 합니다.

```bash
# systemd에 변경된 fstab 정보를 갱신시킴
systemctl daemon-reload

# fstab에 등록된 모든 장치 마운트 시도 (에러 검증용)
mount -a

# fstab 내용의 종합적인 유효성 및 문법 검사 도구
findmnt --verify
```

위의 명령어에서 아무 에러가 출력되지 않는다면 안전하게 시스템을 재부팅할 수 있습니다.
