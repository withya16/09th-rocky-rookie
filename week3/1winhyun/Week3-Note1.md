# RHEL 10 기반 디스크 및 파티션 관리 학습 노트

> Red Hat Enterprise Linux (RHEL) 10 환경에서 효율적인 저장소 관리를 위해 숙지해야 할 파티셔닝 기술, 파일 시스템 관리 전략 및 영구 마운트 설정 방법을 다룹니다.

---

## 1. 파티션 개요 및 테이블 구조 비교

**파티션(Partition)** 이란 하나의 물리적 디스크를 논리적으로 나눈 독립된 영역으로, 운영체제는 각 파티션을 별도의 저장 장치처럼 인식하고 관리합니다.

디스크 파티셔닝은 이러한 파티션을 생성하여 물리적 저장 장치를 독립적인 논리적 영역으로 분할하는 프로세스입니다.

### 1.1. 디스크 파티셔닝의 이점

- **관리 효율성(Management Efficiency)**: 물리적 볼륨에 대한 관리 소홀을 방지하고 데이터 용도별로 공간을 효율적으로 배분합니다.
- **보안 및 조직화(Security & Organization)**: 시스템 파일과 사용자 데이터를 분리하여 특정 파티션의 손상이 전체 시스템으로 확산되는 것을 방지합니다.
- **백업 최적화(Backup Optimization)**: 핵심 데이터가 담긴 파티션만 선택하여 백업함으로써 시간과 저장 공간을 절약합니다.

### 1.2. MBR vs GPT 상세 기술 비교

RHEL 10은 최신 표준인 GPT를 지향하며, 두 방식은 구조적으로 큰 차이가 있습니다.

#### MBR (Master Boot Record)

1983년에 도입된 전통적인 파티션 테이블 방식입니다. 디스크의 첫 번째 섹터(512바이트)에 부트로더 코드와 파티션 테이블 정보를 함께 저장합니다. 파티션 테이블 영역이 64바이트로 고정되어 있어 최대 4개의 주(Primary) 파티션만 정의할 수 있으며, 그 이상이 필요한 경우 확장(Extended) 파티션 안에 논리(Logical) 파티션을 생성하는 방식으로 우회합니다. 주소 지정에 32비트 LBA를 사용하므로 512바이트 섹터 기준 최대 2TiB까지만 인식할 수 있습니다.

#### GPT (GUID Partition Table)

UEFI 표준의 일부로 정의된 최신 파티션 테이블 방식입니다. 각 파티션을 GUID(Globally Unique Identifier)로 식별하며, 64비트 LBA를 사용하여 사실상 무한에 가까운 디스크 크기를 지원합니다. 파티션 테이블을 디스크의 시작과 끝 양쪽에 이중으로 저장(Primary GPT + Secondary GPT)하여 한쪽이 손상되더라도 복구가 가능합니다. 또한 LBA 0에 Protective MBR을 배치하여 MBR 전용 도구가 GPT 디스크를 실수로 덮어쓰는 것을 방지합니다.

| 특징 | MBR (Master Boot Record) | GPT (GUID Partition Table) |
|---|---|---|
| **최대 파티션 개수** | 주(Primary) 4개 또는 주 3개 + 확장(Extended) 1개 (논리 파티션은 연결 리스트 구조로 MBR 스펙상 개수 제한 없음, 커널 제한 적용) | 기본 128개 (공간 추가 할당을 통해 확장 가능) |
| **최대 디스크/파티션 크기 (512b)** | 2 TiB | 8 ZiB (Zebibyte) |
| **최대 디스크/파티션 크기 (4k)** | 16 TiB | 64 ZiB |
| **구조적 특징** | 디스크 시작 부분 고정 섹터 사용 | LBA(Logical Block Addressing) 사용 |
| **복구 지원 (Redundancy)** | 없음 (단일 지점 실패 위험) | 디스크 끝에 보조(Secondary) GPT 테이블 제공 |
| **보안 및 호환성** | 해당 사항 없음 | LBA 0에 Protective MBR을 두어 구형 도구로부터 보호 |

> **참고:** 리눅스 커널은 단일 디스크에서 최대 60개의 파티션까지만 액세스하도록 제한합니다.

### 1.3. RHEL 10의 주요 변화

- **GPT 기본값 적용**: RHEL 10은 BIOS와 UEFI 시스템 모두에서 GPT를 기본 파티션 테이블로 사용합니다.
- **MBR 설치 파라미터**: 신규 설치 시 MBR 형식을 강제해야 하는 특수 상황에서는 `inst.disklabel=mbr` 부트 파라미터를 사용해야 합니다.

## 2. 디스크 명명 규칙 및 영구 식별자

장치 이름은 하드웨어 연결 순서에 따라 변할 수 있으므로, 안정적인 시스템 운영을 위해서는 영구 식별자 이해가 필수적입니다.

### 2.1. 장치 명명 체계 (`/dev/xxyN`)

- **`/dev/`**: 모든 장치 파일이 위치하는 디렉토리입니다.
- **`xx` (장치 유형)**: `sd`는 SCSI/SATA/SAS 디스크, `nvme`는 NVMe 장치를 의미합니다.
- **`y` (장치 순서)**: 장치 인식 순서에 따라 알파벳이 부여됩니다 (예: `sda`, `sdb`). 26개를 초과하면 `sdaa` 식으로 명명됩니다.
- **`N` (파티션 번호)**: 파티션 순서 숫자입니다. MBR의 경우 1~4는 주/확장, 5번부터는 논리 파티션입니다.
- **NVMe 예시**: `/dev/nvme0n1p1` (숫자로 끝나는 NVMe 장치는 파티션 구분자 `p`를 사용)

### 2.2. 영구 속성 (Persistent Attributes)

- **UUID (Universally Unique Identifier)**: 파일 시스템 생성 시 부여되는 128비트 고유 식별자입니다. 마운트 설정 시 가장 권장되는 방식입니다.
- **WWN (World Wide Name)**: 저장 장치 하드웨어 자체에 부여된 고유 식별자입니다.

### 2.3. 마운트 포인트 (Mount Points)

특정 파티션의 저장 공간을 파일 시스템 트리 내의 특정 디렉토리와 연결하는 것입니다.

> **예:** `/dev/sda5`가 `/usr`에 마운트되면, `/usr/bin/` 아래의 파일들은 실제 물리적으로 `sda5` 파티션에 저장됩니다.

## 3. 파티션 관리 도구 및 실무 활용

### 3.1. parted 활용 가이드 (고급 인터페이스)

`parted`는 MBR과 GPT를 모두 지원하며, 대화형 및 비대화형 실행이 가능합니다.

- **파티션 테이블 생성**: `mklabel gpt` (또는 MBR용 `mklabel msdos`).
- **파티션 생성**: `mkpart` 명령어를 사용합니다.

> **주의:** `parted`에서 제공하는 `fs-type` 인수는 실제 파일 시스템을 포맷하는 것이 아니라, **GPT의 파티션 유형 GUID(또는 MBR의 플래그)**를 설정하는 역할만 수행합니다.

- **플래그 설정**: `set <번호> <플래그> on` 명령을 사용합니다.
  - **주요 플래그**: `boot` (MBR용), `esp` (UEFI용), `lvm`, `raid`.
  - **parted 3.4 이후 추가 플래그**: `chromeos_kernel`, `bls_boot`.

### 3.2. fdisk 및 gdisk

- **fdisk**: MBR과 GPT를 모두 지원합니다. MBR 디스크에서는 16진수 코드(예: `82`: Swap, `83`: Linux)로, GPT 디스크에서는 GUID 기반 파티션 타입으로 구분합니다.
- **gdisk**: GPT 전용 도구로, `fdisk`와 유사한 인터페이스를 제공하면서 GPT의 구조적 이점을 모두 활용합니다.

## 4. 파일 시스템 선택 및 관리 전략

**파일 시스템(File System)** 이란 파티션 위에서 데이터를 체계적으로 저장하고 관리하기 위한 구조입니다. 파티션이 빈 땅이라면, 파일 시스템은 그 위에 세운 창고의 선반 배치와 같습니다. 파일 시스템이 없으면 운영체제는 파티션에 데이터를 읽고 쓸 수 없으며, `mkfs` 명령어로 포맷하여 생성합니다.

#### XFS

Silicon Graphics(SGI)가 1993년에 개발한 64비트 저널링 파일 시스템입니다. 할당 그룹(Allocation Groups) 구조를 통해 여러 스레드가 동시에 I/O를 병렬 처리할 수 있어 대용량 파일과 높은 동시성 환경에 강합니다. RHEL 7부터 기본 파일 시스템으로 채택되었습니다.

#### ext4

리눅스의 전통적인 ext 계열(ext2 → ext3 → ext4)의 최신 버전으로, 2008년에 안정화된 저널링 파일 시스템입니다. 오랜 검증을 거쳐 안정성이 높고, 소규모 파일 처리에 효율적이며, XFS와 달리 파일 시스템 크기를 축소(Shrink)할 수 있는 유연성을 제공합니다.

### 4.1. XFS vs ext4 특징 비교

| 기능 | XFS | ext4 |
|---|---|---|
| **RHEL 10 기본값** | 권장 (Default) | 선택 가능 |
| **최대 파일 시스템 크기** | 8 EiB | 1 EiB |
| **최대 단일 파일 크기** | 8 EiB | 16 TiB |
| **Reflink 지원** | 예 (Copy-on-Write) | 아니요 |
| **온라인 확장 (Grow)** | 가능 | 가능 |
| **축소 (Shrink)** | 원천 불가능 | 가능 (오프라인 전용) |
| **아키텍처 강점** | 할당 그룹(Allocation Groups) 기반 병렬 I/O | 소규모 파일 처리 효율성 및 안정성 |

### 4.2. 워크로드별 추천 가이드

- **XFS 선택**: 대규모 엔터프라이즈 워크로드, 고성능 데이터베이스, 비디오 스트리밍. 특히 Reflink 기능을 통해 데이터 복제 없이 즉각적인 **VM 클로닝(Cloning)** 과 스냅샷 기반 워크로드를 지원합니다.
- **ext4 선택**: 파티션 크기 축소(Shrink)가 예상되는 환경, 100GB 미만의 소규모 파티션, 또는 부팅 파티션.

### 4.3. 고급 저장소 기술: VDO (Virtual Data Optimizer)

LVM-VDO는 인라인 중복 제거(Deduplication)와 압축(Compression)을 제공합니다.

- **공간 절약**: 가상 머신 이미지의 경우 최대 10:1 이상의 절약 효과를 제공합니다.
- **하드웨어 요구사항**: VDO 모듈은 기본 370 MB에 추가로 물리 저장소 1 TB당 약 268 MB의 RAM을 요구하며, UDS 중복 제거 인덱스 구성 방식(Dense/Sparse)에 따라 추가 RAM이 필요합니다.

## 5. 영구 마운트 설정 및 관리

### 5.1. `/etc/fstab` 구성 (6개 필드)

```
[장치 식별자(UUID)] [마운트 포인트] [파일 시스템 타입] [마운트 옵션] [덤프 순서] [파일 시스템 체크 순서]
```

### 5.2. 실무 최적화 팁

- **성능 향상**: `noatime` 옵션을 사용하면 파일 접근 시마다 발생하는 메타데이터 쓰기를 줄여 I/O 성능을 높일 수 있습니다.
- **포맷 속도**: XFS 포맷 시 `-K` 옵션을 사용하면 discard(TRIM) 과정을 건너뛰어 대용량 디스크의 포맷 시간을 획기적으로 단축할 수 있습니다.
- **VDO/SSD**: 마운트 옵션에 `discard`를 추가하여 온라인 TRIM을 활성화함으로써 삭제된 블록을 즉시 재사용하도록 설정하십시오.

### 5.3. 검증 절차

설정 변경 후 반드시 다음 명령으로 오타 및 오류를 확인하십시오.

```bash
# 모든 설정을 검증하고 실제 마운트 수행
mount -a
```

## 6. RHEL 10 스터디 요약 및 주의사항

> **파티션 테이블 생성(`mklabel`)과 파일 시스템 포맷(`mkfs`) 작업은 해당 장치의 데이터를 파괴합니다. `mkpart`는 파티션 경계만 정의하므로 직접적인 데이터 파괴는 발생하지 않으나, 잘못된 파티션 설정은 기존 데이터 접근을 불가능하게 할 수 있습니다. 실행 전 장치 이름(`/dev/sdX`)을 최소 3회 이상 확인하십시오.**

- **XFS 축소 불가**: XFS 파일 시스템은 온라인 상태에서 용량을 늘릴 수 있으나, 이미 생성된 파일 시스템의 크기를 줄이는 것(Shrink)은 불가능합니다. 초기 용량 설계 시 이 점을 반드시 고려하십시오.
- **BIOS Boot 파티션 (1MB)**: GPT 파티션 테이블을 사용하는 시스템이 구형 BIOS 모드(또는 CSM)로 부팅할 경우, GRUB 부트로더의 코드를 저장하기 위한 **1MB 크기의 'BIOS Boot Partition'(`bios_grub` 플래그)**이 반드시 필요합니다.
- **백업 권장**: 모든 디스크 구조 변경 작업 전에는 반드시 전체 데이터 백업을 수행하고 복구 계획을 수립하십시오.
