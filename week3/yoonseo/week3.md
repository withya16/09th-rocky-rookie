# 1. 디스크와 파티션 구조 이해

### 디스크

시스템에 장착되어 있는 물리적 저장 장치 

### 파티션

디스크의 공간을 논리적으로 분할하는 공간 , 하나의 물리적 디스크 저장 장치를 여러 독립적인 공간으로 구분 가능해서 서로 다른 OS사용도 가능 

- 핵심 목적은 저장공간을 역할별로 나누기 위함 ex) 시스템 영역과 데이터 영역 분리하기
- 파티션을 마운트하면 그 파티션 위에 만들어진 파일 시스템을 디렉터리에 연결해서 쓰는 것
`/dev/sda5`가 `/usr`에 마운트되면 `/usr` 아래 파일들은 물리적으로 그 파티션에 저장되는 것

### **파티션 테이블**

파티션을 쓰려면 디스크에 어떤 형식의 파티션 테이블을 둘지 정해야함. 이 방식이 2가지 존재 

- **GPT** : 최대 128개의 파티션 지원, 매우 큰 디스크 다룰 수 있어서 현대 시스템의 표준
- **MBR** : 최대 4개 주 파티션 , 더 만들려면 확장 파티션과 논리 파티션 구조를 써야함
    - 각 파티션 테이블 속 파티션의 중요한 특성
        - 파티션이 시작되고 끝나는 디스크의 지점
        - 하나의 파티션만 활성으로 플래그 지정할 수 있는 파티션 상태
        - 파티션 유형
- `실제 관리 흐름 : 디스크 확인 → 파티션 테이블 만들기 → 그 위에 파티션 생성 → 파일 시스템 만들기 → 원하는 디텍터리에 마운트해서 사용하기`
- `포맷`의 의미 : 이 파티션을 ext4나 xfs 같은 파일 시스템으로 초기화해서, 파일을 저장할 수 있게 만든다 
단순히 나누어놓은 파티션이 아니라 그 안에 초기세팅을 해두어야하는것 그래야 파일 시스템이 됨

### 파티션 생성 후 포맷 및 마운트

1. 현재 상태 확인 
    
    ```sql
    sudo parted /dev/vdb print
    ```
    
2. parted 실행 : sudo parted /dev/vdb
    
    ```sql
    sudo parted /dev/vdb
    ```
    
3. GPT 파티션 테이블 생성 
    
    ```sql
    mklabel gpt 
    ```
    
4. 파티션 생성 
    
    ```sql
    mkpart primary xfs 1MiB 2GiB 
    ```
    
    - 파티션 이름/종류: primary
    - 파일시스템 타입 표기: xfs
    - 시작 위치: 1MiB
    - 끝 위치: 2GiB
    
5. 포맷 
    
    ```sql
    sudo mkfs.xfs /dev/vdb1
    ```
    
6. 마운트 
    
    ```sql
    sudo mkdir /mnt/testdisk
    sudo mount /dev/vdb1 /mnt/testdisk
    ```

# 3. LVM 개념과 볼륨 구성

### LVM 개념

Logical Volume Manager, 조금 더 유연하게 스토리지 볼륨을 관리하기 위한 방식 

기존 방식보다 크기 조절, 이동, 확장, 구성 변경이 훨씬 유연함 

기존에는 파티션 위에 파일시스템 만들고 마운트해서 사용했지만 이 방식은 나중에 저장 공간이 부족해졌을 때 불편함 다른 디스크에 남는 공간이 있어도 파티션으로 구분되어있는 이상 합치기는 어려움. But LVM은 여러 물리 장치를 묶어서 풀처럼 다루고 거기서 필요한 만큼 논리 볼륨을 꺼내서 사용함 

- `디스크/파티션을 PV로 만듦`
- `PV를 묶어서 VG를 만듦`
- `VG 안에서 LV를 하나 만듦`

### 1. PV

- Physical Volume
- LVM이 사용할 물리 저장소
- 파티션 또는 전체 디스크일 수 있음
- `pvcreate /dev/sdb1` 
아직 파티션 or 디스크이기만 한 공간에 이 공간은 LVM 공간으로 쓸게 ! 라고 선언한 상태

### 2. VG

- Volume Group
- PV 를 여러개로 묶어서 만든 저장 공간 풀

ex) `/dev/sdb1` 50G , `/dev/sdc1` 100G 이 두개를 PV로 만든 후 하나의 VG로 묶으면 총 150G짜리 저장공간 풀처럼 다룰 수 있음 `vgcreate vgdata /dev/sdb1 /dev/sdc1`

- 처음부터 그냥 PV 150G로 만들면 되는거 아닌가 ?
    
    처음엔 50G만 있었는데 나중에 공간이 부족해져서 100G 디스크를 새로 달 수 있음 
    
    그때 일반 파티션이면 새 디스크는 그냥 별개의 /dev/sdc1 일 뿐인데, LVM이면 새 디스크를 PV로 만든 뒤 기존 VG에 추가해서 저장공간 풀 자체를 키울 수 있음 
    
    - 처음: 50G
    - 나중: 100G 추가
    - 결과: 논리적으로는 150G 풀처럼 확장 가능

### 3. LV

- Logical Volume
- LV는 실제로 사용하게 되는 논리 디스크
- VG라는 큰 공간 풀에서 원하는 크기만큼 잘라 만든 가상 디스크가 LV !

ex) VG가 150G 있다면  그 안에서 50G짜리 `lvhome` , 80G짜리 `lvdata` ,20G짜리 `lvbackup` 이런 식으로 나눠서 쓸 수 있음 

- 대표 명령어 : `lvcreate -L 50G -n lvdata vgdata` 
”vgdata 안에서 50G를 꺼내서, lvdata라는 이름의 LV를 만든다”

### 볼륨 구성

단순히 LV를 만드는거에서 끝나는게 아니라, LV를 그냥 평범하게 만들지, 성능 좋게 만들지, 장애 대비되게 만들지, 필요할 때만 공간 쓰게 만들지

- 가장 기본 : **`linear 볼륨`** (아무 옵션 없는 일반적인 LV)
    - VG 안의 공간을 순서대로 할당해서 LV를 만듦
    - 특별한 성능 분산도 없고
    - 복제도 없고
    - 그냥 가장 기본적인 논리 볼륨
    
    ex) PV1= 50G, PV2 = 50G, VG= 100G 인 상태에서 30G LV 만들면 보통 VG 안의 남는 공간을 **순서대로** 써서 `lvdata`를 만드는 것 
    
- **`Striped 볼륨`** : 속도를 위해서 나눠서 저장 
두 개 이상의 장치에 데이터를 스트라이프 방식으로 분산 저장해서 throughtput을 높이는 것!
    - 일반 LV: 파일 A를 디스크1에 계속 씀
    - Striped LV : 파일 A의 데이터를 일부는 디스크1, 일부는 디스크2, 일부는 디스크1, 일부는 디스크 2 이런식으로 번갈아서 저장
    - striped LV를 만들면 데이터가 두 디스크에 나뉘어 저장돼서 읽고 쓰는 작업을 병렬로 처리할 수 있어 성능 향상에 유리함
    - 대용량 순차 읽기/쓰기 , 성능이 중요한 경우 좋음
    - But 데이터가 여러 디스크에 걸쳐있어서 하나라도 망가지면 전체 LV가 깨질 수 있음 , 성능은 좋을 수 있지만 안정성은 별도 보장되지 않음
- **`RAID 볼륨`** : 장애 대비 + 성능 설계
    
    중복성과 성능을 위해 여러 디스크를 활용하는 방식 , striped는 “나눠 저장”이고, RAID는 “나눠 저장 + 보호 규칙”까지 포함한 방식 
    
    - RAID 0
        - 데이터를 여러 디스크에 분산 저장
        - 빠름
        - 하지만 하나 고장 나면 전체 손실 가능
    - RAID 1
        - 데이터를 복제해서 저장
        - 예: 디스크 A에 쓰면 디스크 B에도 똑같이 씀
        - 한 디스크가 고장 나도 다른 디스크에 데이터가 남음
    - RAID 5/6
        - 데이터를 여러 디스크에 나눠 쓰면서
        - 일부는 패리티(복구 정보)를 저장
        - 디스크 일부가 고장 나도 복구 가능
    - RAID 10
        - 분산 저장 + 복제 조합
- **`Thin Provisioning`** : 미리 다 안주고 쓸 때만 진짜 줌
    
    실제 물리 저장공간보다 더 큰 논리 볼륨을 만들 수 있게 하고, 실제 사용량에 따라 공간을 동적으로 할당하는 것 
    
    ex) 
    
    - 실제 디스크 총합: 200G
    - thin pool 생성
    - 사용자에게
        - LV A = 100G
        - LV B = 100G
        - LV C = 100G
        이렇게 총 300G를 논리적으로 줄 수도 있음
    
    처음부터 300G를 다 쓰는 게 아니라, 실제 파일이 저장될 때만 물리 공간을 쓰기 때문
    
    비행기, 호텔에서의 오버부킹 같은 느낌 
    
    - 저장공간 활용 효율이 좋음
    - 가상 머신, 테스트 환경, 컨테이너 환경에 유용
    - But 실제 물리 공간이 다 찼는데 논리 볼륨은 더 큰 상태일 수 있음
- **`Snapshot`** : 특정 시점 상태를 잠시 얼려두기 
snapshot = 특정 시점의 논리 볼륨 복사본
    - 백업과는 다른 개념,  “모든 데이터를 처음부터 전부 복사해서 새로 하나 더 만드는 것”이라기보다, 보통 변경된 블록을 기준으로 관리해 효율적으로 시점 복사를 제공한다는 개념
    - 백업 전에 일관된 상태 확보
    - 실수로 변경했을 때 비교
    - 점검/업데이트 전 상태 보관
    
    ex) 
    
    - 지금 `lvdata` 안에 중요한 데이터가 있음
    - 지금 시점 상태를 스냅샷으로 잡아 둠
    - 그 뒤 원본 LV에서 파일이 바뀌어도
    - 스냅샷은 “그 시점의 모습”을 기준으로 유지
- **`Caching`**: 느린 디스크 앞에 빠른 디스크 붙이기
    - **느린 저장소에 대해 더 빠른 저장소를 캐시로 사용**하는 기능
    
    ex) 
    
    - 실제 데이터 본체는 느린 HDD에 있음
    - 자주 읽는 데이터는 SSD 캐시에 저장

# 4. 파일 시스템 생성 및 점검

### 파일 시스템

단순히 디스크 공간이 아니라 그 공간 안에 파일과 디렉터리를 어떤 규칙으로 저장하고 관리할지 정하는 형식 

- 파티션 생성: 디스크 공간을 나눔
- 파일 시스템 생성: 그 나눈 공간을 실제로 파일 저장 가능한 구조로 초기화함
- 마운트: 그 파일 시스템을 OS의 특정 디렉터리에 연결해서 사용 가능하게 함

파티션 있어도 파일 시스템 생성 안하면 파일을 저장하는 공간으로 사용하지 못 함 

### XFS

RHEL 10에서는 XFS가 기본이다. XFS는 대용량 파일 시스템과 큰 파일 처리에 강하고, 메타 데이터 저널링, 빠른 복구 , 온라인 확장 기능 지원해준다. 특히 서버나 대용량 스토리지 환경에서 성능과 확장성이 좋은 편 (ext4 : 여전히 많이 쓰이는 파일 시스템 중 하나 )

- 파일 시스템 생성
    
    ```sql
    #xfs의 경우 
    mkfs.xfs /dev/sdb1 
    
    #ext4의 경우 
    mkfs.ext4 /dev/sdb1
    ```
    
    - 기본 옵션은 일반적인 용도에서 보통 최적
    - 기존 파일 시스템이 이미 있는 장치를 덮어쓸 때는 `f` 필요
    - RAID 장치면 stripe geometry를 고려할 수 있음
- 이후 바로 마운트 해줘야함 : `mount /dev/sdb1 /mnt/data` : 장치를 `/mnt/data`에 연결
    - `df -h` : 실제 마운트되었는지 용량까지 확인

### 파일 시스템 생성 전 필요한 것

파일 시스템은 **블록 장치** 위에 만듦 

- 일반 파티션: `/dev/sdb1`
- LVM 논리 볼륨: `/dev/vgname/lvname`
- RAID 장치
- 경우에 따라 디스크 전체

```
lsblk
blkid
```

- `lsblk`: 디스크/파티션 구조 확인
- `blkid`: 파일 시스템 타입, UUID 확인

### `fsck` : 파일 시스템 점검

평소에는 멀쩡히 쓰다가도 갑작스러운 전원 꺼짐, 강제 종료, 저장장치 오류, 잘못된 스토리지 작업같은 이유로 파일 시스템 메타 데이터가 꼬일 수 있어서 이때 점검 도구를 통해 파일 시스템 구조의 일관성을 확인하고 복구하는 것 

- 파일 시스템 점검이 필요한 상황
    - 시스템 부팅 실패
    - 특정 디스크의 파일이 손상됨
    - 파일 시스템이 불일치 때문에 read-only로 바뀜
    - 파일에 접근할 수 없음
- 메타 데이터 일관성을 보장하는 도구
- Not 실제 데이터 복구 도구

### `xfs_repair` : XFS 점검/복구 방식

xfs는 비정상 종료가 있어도 mount 시점에 저널을 재생해서 일관성을 맞추는 방식이 기본임xfs_repair는 부팅 때 자동으로 도는 일반 fsck 방식이 아니라 필요할 때 관리자가 수동으로 사용함 dirty log가 있는 상태에서는 바로 수리할 수 없고 먼저 마운트/언마운트 등을 통해 로그 상태를 정리해야함 

- “파일 시스템이 스스로 말이 안 되는 상태가 된 것”을 고치는 도구
- XFS 점검
    
    ```sql
    sudo umount /dev/sdb1
    sudo xfs_repair -n /dev/sdb1
    ```
    
    - `n` : 실제 수정 없이 dry run
    - 어떤 문제가 있는지 먼저 확인할 때 사용
- XFS 실제 복구
    
    ```sql
    sudo umount /dev/sdb1
    sudo xfs_repair /dev/sdb1
    ```
    
- **정상적이지 않은 종료** → 일단 mount 시 저널 재생
- 그래도 문제 있음 → `xfs_repair -n`
- 실제 복구 필요 → `xfs_repair`

### XFS 와 Ext4 명령어 정리

- **생성**
    - XFS: `mkfs.xfs`
    - ext4: `mkfs.ext4`
- **점검/복구**
    - XFS: `xfs_repair`
    - ext4: `e2fsck`
- **크기 조정**
    - XFS: `xfs_growfs`
    - ext4: `resize2fs`

# 5. 마운트·자동 마운트 설정(/etc/fstab) 실습

### 마운트

- 파티션으로 만든 저장 공간을 어떤 디렉터리에 연결하는 동작

### 수동 마운트

```sql
sudo mount /dev/sdb1 /data
```

- 직접 명령어 쳐서 연결 But 현재 부팅된 세션에만 적용되어서 재부팅하면 사라짐

### 자동마운트

```sql
/etc/fstab
```

- 부팅할 때마다 자동으로 붙게 설정하는 것
- `/etc/fstab`가 왜 중요할까?
    
    시스템이 부팅될 때 `systemd-fstab-generator`가 `/etc/fstab` 내용을 읽고 mount unit으로 변환해서 마운트를 처리해서  `/etc/fstab`에 잘못 쓰면 **부팅 문제**가 생길 수 있음 ! 
    
- `/etc/fstab` 한 줄 구조
    
    ```
    UUID=ea74bbec-536d-490c-b8d9-5b40bbd7545b /boot xfs defaults00
    장치식별자   마운트지점   파일시스템종류   옵션   dump   fsck순서
    ```
    
    - dump 0 : 백업 도구 `dump` 관련 설정. 보통 0
    - fsck 순서 0 : 부팅 시 파일 시스템 검사 순서, XFS는 일반적으로 fsck 대상으로 다루지 않아서 0이 흔함.
    - UUID 쓰는 이유 :
        - `/dev/sdb1` 같은 이름은 장치 인식 순서가 바뀌면 달라질 수 있음
        - UUID는 그 파일 시스템 고유 번호라서 더 안정적임
- 실습
    
    ```
    lsblk--fs /dev/sdb1
    sudomkdir-p /data
    sudo mount /dev/sdb1 /data
    df-Th |grep /data
    
    sudovi /etc/fstab
    ```
    
    추가:
    
    ```
    UUID=1234abcd-56ef-78gh-90ij-1234567890ab /data xfs defaults00
    ```
    
    그 다음:
    
    ```
    sudo systemctl daemon-reload
    sudo umount /data
    sudo mount /data
    df-Th |grep /data
    sudo mount-a
    ```
    
    재부팅 후 최종 확인:
    
    ```
    reboot
    ```
    
    다시 접속해서: 
    ```
    df-Th |grep /data
    ```
