# 리눅스 프로세스 관리 및 자원 제어

---

## 1부: 리눅스 프로세스의 이해와 상태 모니터링

**프로세스(Process)** 는 디스크에 저장된 정적인 프로그램(`.jar` 파일, 바이너리 실행 파일 등)이 메모리에 적재되어 CPU에 의해 실행되고 있는 **동적인 상태**를 의미합니다.

---

### 1.1 프로세스의 기본 구조와 계층

- **PID (Process ID)**: 커널이 모든 프로세스에 부여하는 고유 번호
- **PPID (Parent Process ID)**: 자신을 생성한 부모 프로세스의 PID

> **프로세스 생성 원리:**
> 부모 프로세스가 `fork()` 시스템 콜로 자신의 복사본을 만들고,
> `exec()`를 통해 새로운 프로그램으로 덮어쓰는 방식으로 생성됩니다.

#### 프로세스 상태 (State)

| 상태 | 기호 | 설명 |
|------|:----:|------|
| **Running/Runnable** | `R` | CPU에서 실행 중이거나 실행 대기열(Run Queue)에서 대기 중 |
| **Interruptible Sleep** | `S` | 파일 I/O, 네트워크 응답 등 특정 이벤트가 발생할 때까지 수면 상태. 대부분의 백엔드 데몬이 이 상태 |
| **Uninterruptible Sleep** | `D` | 디스크 I/O 등 커널 수준 하드웨어 작업 중 — `kill` 시그널을 받아도 죽지 않음. **이 상태가 지속되면 시스템 부하 급증** |
| **Zombie** | `Z` | 실행이 종료되어 자원은 반환했으나, 부모 프로세스가 종료 상태를 회수(`wait()`)하지 않아 프로세스 테이블(PID)만 차지하는 찌꺼기 상태 |

---

### 1.2 `ps` (Process Status): 정적 스냅샷 조회

현재 실행 중인 프로세스의 **찰나의 상태**를 캡처하여 보여주는 도구입니다.

| 명령어 | 계열 | 주요 출력 필드 |
|--------|------|--------------|
| `ps -ef` | System V | UID, PID, PPID, C, STIME, TTY, TIME, CMD |
| `ps aux` | BSD | CPU 사용률(`%CPU`), 메모리 사용률(`%MEM`) 포함 |

**심화 활용 — Spring Boot 프로세스 찾기:**

```bash
ps -ef | grep java
```

JVM 위에서 구동 중인 Java 프로세스의 PID, 힙 메모리 옵션(`-Xms`, `-Xmx`), 실행된 `.jar` 경로를 한눈에 파악할 수 있습니다.

---

### 1.3 `top`: 실시간 동적 모니터링

`ps`가 **사진**이라면, `top`은 **동영상**입니다.
시스템의 전반적인 요약 정보와 자원 점유율이 높은 프로세스들을 실시간(기본 3초 갱신)으로 정렬하여 보여줍니다.

#### Load Average (부하 평균) 해석

```
load average: 1.05, 0.70, 0.50
               │      │      │
              1분    5분    15분  평균 대기 프로세스 수
```

> **기준점은 CPU 코어 수입니다.**
> 4코어 서버에서 Load Average가 `4.0`이면 CPU가 100% 가동 중.
> `4.0`을 초과하면 프로세스가 큐에 쌓이며 **서비스 지연이 발생**합니다.

#### CPU 상태 지표

| 지표 | 설명 | 높을 때 조치 |
|------|------|------------|
| `us` (user) | 사용자 공간(백엔드 앱, DB 등) CPU 사용률 | 애플리케이션 로직 최적화 |
| `sy` (system) | 커널 공간(시스템 콜, 네트워크 I/O 처리 등) CPU 사용률 | 시스템 콜 횟수 점검 |
| `wa` (iowait) | 디스크/네트워크 I/O를 기다리며 CPU가 유휴 상태인 비율 | **CPU/RAM 증설이 아닌 SSD 교체 또는 쿼리 튜닝 필요** |
| `id` (idle) | 유휴 상태(남아도는) CPU 비율 | - |

## 2부: 프로세스 제어 (시그널과 우선순위)

### 2.1 `kill`: 시그널(Signal)을 통한 프로세스 제어

`kill`은 단순히 프로세스를 '죽이는' 명령어가 아니라,
프로세스에 **특정 소프트웨어 인터럽트(Signal)를 보내는 통신 명령어**입니다.

| 시그널 | 번호 | 이름 | 설명 |
|--------|:----:|------|------|
| **SIGTERM** | `15` | Terminate | 기본값. "안전하게 하던 작업을 마무리하고 종료해라"고 권고. Spring Boot는 이 신호를 받으면 DB 커넥션 풀을 닫고 처리 중인 HTTP 요청을 완료한 뒤 스스로 종료 **(데이터 유실 방지의 핵심)** |
| **SIGKILL** | `9` | Kill | 커널이 프로세스를 즉각 강제 삭제. SIGTERM을 무시하거나 좀비 상태에 빠진 프로세스에만 **최후의 수단**으로 사용. 심각한 데이터 손상이나 파일 잠금(Lock) 문제 유발 가능 |
| **SIGHUP** | `1` | Hangup | 데몬 프로세스를 끄지 않고 **설정 파일만 다시 읽어오도록** 지시. Nginx 무중단 설정 리로드 시 활용 |

```bash
kill -15 [PID]    # SIGTERM — 우아한 종료 (권장)
kill -9  [PID]    # SIGKILL — 강제 종료 (최후의 수단)
kill -1  [PID]    # SIGHUP  — 설정 리로드
```

### 2.2 `nice`와 `renice`: CPU 스케줄링 우선순위 조정

여러 프로세스가 동시에 실행될 때, 커널의 스케줄러(CFS)는 **NI (Nice) 값**을 참고하여 어떤 프로세스에 CPU를 먼저 할당할지 결정합니다.

```
-20 ◄──────────────────────────────────────► 19
높은 우선순위                             낮은 우선순위
(안 착함, CPU 독점)    기본값: 0         (착함, CPU 양보)
```

| 명령어 | 설명 |
|--------|------|
| `nice -n 15 [명령어]` | 새 프로세스를 낮은 우선순위로 시작 |
| `renice -n 15 -p [PID]` | 이미 실행 중인 프로세스의 우선순위 변경 |

> **실무 예시:** 로그 압축이나 대규모 백업은 CPU를 많이 소모하므로,
> 메인 백엔드 서비스에 영향을 주지 않도록 우선순위를 낮추어 실행합니다.
> ```bash
> nice -n 15 tar -czvf backup.tar.gz /data
> ```
> **주의:** 음수 값(우선순위 높임)은 `root` 관리자만 부여할 수 있습니다.

## 3부: 작업 스케줄링 자동화 (`cron`과 `at`)

서버 운영 시 주기적인 배치(Batch) 작업이나 정해진 시간의 1회성 작업을 사람의 개입 없이 처리해야 합니다.

---

### 3.1 `cron`: 주기적 스케줄링의 표준

`crond` 데몬은 **1분마다** 스케줄 테이블을 확인하여 조건이 맞는 명령어를 실행합니다.

#### Crontab 작성 구조 (`crontab -e`)

```
# ┌───────────── 분 (0 - 59)
# │ ┌────────────── 시 (0 - 23)
# │ │ ┌─────────────── 일 (1 - 31)
# │ │ │ ┌──────────────── 월 (1 - 12)
# │ │ │ │ ┌───────────────── 요일 (0 - 7) (0과 7은 일요일)
# │ │ │ │ │
# * * * * * /path/to/script.sh
```

**자주 쓰는 표현식:**

| 표현식 | 의미 |
|--------|------|
| `*/5 * * * *` | 5분마다 |
| `0 3 * * *` | 매일 새벽 3시 |
| `0 0 1,15 * *` | 매월 1일과 15일 자정 |
| `0 9 * * 1-5` | 평일(월~금) 오전 9시 |

#### 심화 주의사항

> **절대 경로 원칙:** `cron`은 일반 로그인 쉘의 환경 변수(`PATH` 등)를 읽지 못하는 **제한된 환경**에서 실행됩니다.
> 명령어와 스크립트 내부의 모든 경로는 반드시 **절대 경로**로 작성해야 합니다.

> **로깅 처리:** cron이 실행한 스크립트의 결과는 기본적으로 로컬 메일로 발송됩니다.
> 로그 파일로 남기려면 리다이렉션을 사용합니다.
> ```bash
> # 매일 새벽 3시에 백업 실행 후 성공/실패 로그 기록
> 0 3 * * * /opt/scripts/db_backup.sh >> /var/log/db_backup.log 2>&1
> ```

---

### 3.2 `at`: 1회성 예약 작업

정기적인 배치가 아닌 **단발성 예약** 작업에 사용합니다.

```bash
at 04:00 tomorrow
# 프롬프트가 뜨면 실행할 명령어 입력
> reboot
# Ctrl + D 로 저장 및 종료
```

> 실행된 작업은 큐에서 즉시 사라집니다.

## 4부: cgroup (Control Groups) 기반 자원 제어

프로세스 관리의 궁극적인 지향점은 **격리와 통제**입니다.

> **문제 상황:** Java 프로세스에 메모리 누수(Memory Leak)가 발생하여 서버의 전체 RAM을 모두 점유하면,
> 운영체제의 **OOM(Out of Memory) Killer** 가 무고한 DB나 중요 시스템 프로세스를 무작위로 강제 종료시키는 대참사가 발생합니다.

이를 커널 레벨에서 원천 차단하는 기술이 **cgroup**이며,
**컨테이너(Docker)와 쿠버네티스(Kubernetes)** 가 파드(Pod)의 CPU/메모리 리미트를 제어하는 근간 기술입니다.

---

### 4.1 cgroup의 핵심 서브시스템(Controller)

| 서브시스템 | 제어 대상 | 설명 |
|-----------|----------|------|
| **`cpu`** | CPU 시간 비율 | 특정 프로세스 그룹이 사용할 수 있는 CPU 할당량 제한 (예: 코어 1개 분량 이상 불가) |
| **`memory`** | RAM 및 스왑 사용량 | 최대 메모리 사용량 제한. 임계치 초과 시 해당 cgroup 내 프로세스만 정확히 타겟팅하여 OOM Kill → **시스템 전체 장애로 번지는 것을 방지** |
| **`blkio`** | 디스크 I/O 속도 | 디스크 읽기/쓰기 속도(IOPS, Bandwidth) 제한 — 특정 앱의 I/O 대역폭 독점 방지 |

---

### 4.2 systemd와 cgroup의 결합 (실무 적용)

최신 리눅스 환경에서는 cgroup을 직접 제어하기보다, **systemd `.service` 파일을 통해** 우아하게 자원을 제한합니다.

```ini
# /etc/systemd/system/backend.service

[Service]
ExecStart=/usr/bin/java -jar /app/backend.jar

# cgroup을 통한 자원 제어
CPUQuota=50%
MemoryHigh=2G
MemoryMax=2.5G
```

| 지시어 | 설명 |
|--------|------|
| `CPUQuota=50%` | 이 프로세스(및 하위 스레드)는 단일 CPU 코어의 50% 이상 절대 사용 불가 |
| `MemoryHigh=2G` | 메모리 사용량이 2GB에 도달하면 커널이 공격적으로 스왑 아웃(메모리 회수)을 시도하여 쓰로틀링 |
| `MemoryMax=2.5G` | **최후의 방어선.** 이 수치를 초과하는 순간 커널이 이 서비스만 즉시 OOM Kill → 다른 인프라(DB 등) 보호 |

```
메모리 사용량 증가
      │
      ▼
   2GB 도달
      │  MemoryHigh — 커널이 스왑 아웃으로 회수 시도 (쓰로틀링)
      ▼
   2.5GB 초과
      │  MemoryMax  — 해당 서비스만 OOM Kill (시스템 전체 보호)
      ▼
 다른 서비스 정상 운영 유지
```
