# RHEL 10 시스템 관리 핵심 정리

## 1. 패키지 설치·업데이트·삭제 (DNF5 활용)

RHEL 10의 가장 큰 변화 중 하나는 기존 파이썬 기반의 DNF4/YUM을 대체하여 **C++로 완전히 재작성된 DNF5**가 기본 패키지 관리자로 도입되었다는 점입니다. 초기 가동 속도가 매우 빠르고 메모리 사용량이 대폭 감소했습니다. 또한 이전 버전에 있던 복잡한 '모듈러리티(Modularity)' 기능이 제거되어, 일반적인 RPM 패키지 방식으로 단순화되었습니다. `dnf` 명령어는 자동으로 `dnf5`로 연결되므로 기존과 같이 사용하셔도 무방합니다.

### 패키지 설치

| 용도 | 명령어 |
|------|--------|
| 단일 패키지 설치 | `dnf install 패키지명` (예: `dnf install nginx`) |
| 자동 예스(yes) 옵션 | `dnf install nginx -y` (중간 확인 과정 생략) |
| 로컬 RPM 파일 설치 | `dnf install /경로/파일.rpm` (의존성까지 자동 해결) |

### 패키지 업데이트

| 용도 | 명령어 |
|------|--------|
| 전체 시스템 업데이트 | `dnf update` 또는 `dnf upgrade` |
| 특정 패키지 업데이트 | `dnf update 패키지명` |
| 보안 관련 패치만 업데이트 | `dnf upgrade --security` |

### 패키지 삭제

| 용도 | 명령어 |
|------|--------|
| 단일 패키지 삭제 | `dnf remove 패키지명` |
| 사용되지 않는 의존성 일괄 정리 | `dnf autoremove` (더 이상 필요 없는 의존성 패키지를 깔끔하게 제거) |

### 작업 되돌리기 (히스토리 롤백)

DNF5는 수행된 작업 이력을 기록합니다.

| 용도 | 명령어 |
|------|--------|
| 트랜잭션 이력 확인 | `dnf history` |
| 특정 작업만 취소 | `dnf history undo <ID>` |
| 특정 시점 이후 모든 변경 되돌리기 | `dnf history rollback <ID>` |

## 2. Repository 구조와 설정 확인

RHEL 10은 시스템의 코어 안정성과 최신 애플리케이션 제공의 균형을 맞추기 위해 리포지토리(저장소)를 명확한 역할로 분리하여 운영합니다.

### Repository의 주요 구조

| 리포지토리 | 역할 |
|------------|------|
| **BaseOS** | 운영체제의 핵심 기능을 구동하는 기초 패키지(커널, systemd 등). 절대적인 안정성을 위해 엄격하게 관리 |
| **AppStream** | 사용자 애플리케이션, 런타임 언어(Python, Node.js 등), 데이터베이스(MariaDB 등) |
| **CodeReady Linux Builder (CRB)** | 개발자를 위한 추가 패키지가 포함된 저장소 |

### Repository 설정 및 확인 방법

리포지토리 설정은 전역 설정 파일인 `/etc/dnf/dnf.conf`와 개별 리포지토리 설정 파일이 위치하는 `/etc/yum.repos.d/` 디렉터리의 `.repo` 파일들로 구성됩니다. RHEL 10부터는 저장소 동기화 시 용량을 많이 차지하는 filelists 메타데이터를 기본적으로 다운로드하지 않아 처리 속도가 훨씬 빠릅니다.

| 용도 | 명령어 |
|------|--------|
| 활성화된 리포지토리 목록 확인 | `dnf repolist` 또는 `dnf5 repo list` |
| 새로운 리포지토리 추가 | `dnf config-manager --add-repo <URL>` |
| 리포지토리 활성화 | `dnf config-manager --set-enabled <저장소_ID>` |
| 리포지토리 비활성화 | `dnf config-manager --set-disabled <저장소_ID>` |

## 3. 사용자 및 그룹 생성·변경·삭제

RHEL 10의 계정 관리는 현대적인 보안 알고리즘을 사용합니다. 특히 비밀번호 해싱 방식이 기존 SHA-512에서 양자 컴퓨터 등 향상된 컴퓨팅 환경의 브루트포스 공격을 방어하는 **메모리 하드 방식의 yescrypt**로 기본 채택되어 보안이 대폭 강화되었습니다.

### 관련 핵심 파일

| 파일 | 설명 |
|------|------|
| `/etc/passwd` | 사용자 계정 기본 정보 |
| `/etc/shadow` | yescrypt 등으로 안전하게 해싱된 비밀번호 정보 (root만 열람 가능) |
| `/etc/group` | 시스템 내 그룹 정보 및 소속 멤버 |

### 사용자 관리 명령어

| 작업 | 명령어 | 비고 |
|------|--------|------|
| 생성 | `useradd <계정명>` | UID/그룹 지정: `useradd -u 1001 -g developers <계정명>` |
| 비밀번호 설정 | `passwd <계정명>` | 설정하지 않으면 로그인 불가 |
| 그룹 추가 | `usermod -aG <그룹명> <계정명>` | **`-a`를 빼면 기존 그룹 목록이 덮어씌워지니 주의** |
| 삭제 | `userdel <계정명>` | 홈 디렉터리까지 삭제: `userdel -r <계정명>` |

> **참고**: RHEL 10 설치 시 생성된 사용자는 기본적으로 관리자 권한(sudo)을 가집니다.

### 그룹 관리 명령어

| 작업 | 명령어 |
|------|--------|
| 생성 | `groupadd <그룹명>` |
| 이름 변경 | `groupmod -n <새이름> <기존이름>` |
| GID 변경 | `groupmod -g <새GID> <그룹명>` |
| 삭제 | `groupdel <그룹명>` |

## 4. 파일 권한과 소유권 관리

리눅스 시스템 데이터 보안의 근간은 파일 시스템 접근 제어입니다. RHEL 10에서도 POSIX 표준 권한 체계를 엄격하게 따릅니다.

### 기본 권한 개념

**소유 계층:**

| 기호 | 의미 |
|------|------|
| `u` | 소유자 (User) |
| `g` | 그룹 (Group) |
| `o` | 기타 사용자 (Others) |
| `a` | 모두 (All) |

**권한 종류:**

| 기호 | 의미 | 숫자 | 디렉터리에서의 의미 |
|------|------|------|---------------------|
| `r` | 읽기 | 4 | 목록 조회 |
| `w` | 쓰기 | 2 | 파일 생성/삭제 |
| `x` | 실행 | 1 | 디렉터리 내부로 접근(`cd`) |

### chmod (권한 변경)

**심볼릭 모드 (직관적 방식)** - 특정 권한을 더하거나 빼는 방식:

```bash
chmod u+x,g-w,o=r file
# 소유자에게 실행 추가, 그룹에서 쓰기 제거, 기타 사용자는 읽기만 가능
```

**8진수 모드 (숫자 방식)** - 권한을 숫자로 합산하여 명시:

```bash
chmod 755 script.sh   # 소유자 7(rwx), 그룹 5(r-x), 기타 5(r-x)
chmod 644 doc.txt     # 소유자 6(rw-), 그룹 4(r--), 기타 4(r--)
```

### chown 및 chgrp (소유권 변경)

| 명령어 | 용도 | 예시 |
|--------|------|------|
| `chown` | 소유자와 소유 그룹을 함께 변경 | `chown nick:developers app.py` |
| `chgrp` | 소유 그룹만 변경 | `chgrp <그룹명> 파일명` |

### 특수 권한 및 기본 권한 (umask)

**umask** - 새로 생성되는 파일과 디렉터리의 기본 권한을 결정하는 마스크 값:

| 사용자 | umask 값 | 디렉터리 기본 권한 | 파일 기본 권한 |
|--------|----------|-------------------|---------------|
| 일반 사용자 | `002` | 775 | 664 |
| root | `022` | 755 | 644 |

### 특수 권한 (고급 접근 제어)

| 권한 | 설정 방법 | 설명 |
|------|-----------|------|
| **SUID** | `chmod u+s` | 프로그램 실행 시 실행자의 권한이 아닌 **파일 소유자**의 권한으로 임시 상승하여 실행. (예: 일반 사용자가 비밀번호를 변경하는 `passwd` 명령어) |
| **SGID** | `chmod g+s` | 디렉터리에 부여할 경우, 새로 만들어지는 파일들이 부모 디렉터리의 그룹 소유권을 강제로 상속받음. 협업 디렉터리에서 필수적 |
| **Sticky Bit** | `chmod +t` | 공용 디렉터리(예: `/tmp`)에서 **오직 파일 소유자만** 해당 파일을 삭제할 수 있도록 보호 |
